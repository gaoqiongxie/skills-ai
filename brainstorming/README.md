## 🎯 什么是结构化头脑风暴？

**Brainstorming** 是一种在编码之前系统思考问题的方法，帮助你：
- 减少返工
- 提前发现风险
- 做出更好的设计决策

> "在编码之前先思考" —— 大多数重构源于早期的设计缺陷

---

## 🔄 4步法流程

```
┌────────────────────────────────────────────────┐
│  步骤1: 明确核心问题                            │
│  ↓                                              │
│  步骤2: 识别约束条件                            │
│  ↓                                              │
│  步骤3: 列举边界情况                            │
│  ↓                                              │
│  步骤4: 生成备选方案                            │
└────────────────────────────────────────────────┘
```

## 📝 使用示例

### 用户说："我要做一个博客系统"

### AI执行4步法：

**步骤1：明确核心问题**
```
🎯 核心需求：
让用户方便地发布和阅读文章

成功标准：
- [ ] 能发布文章
- [ ] 能阅读文章
- [ ] 能评论文章
```

**步骤2：识别约束条件**
```
📋 约束条件：
- 技术栈：Java + MySQL
- 时间：2周
- 团队：2人
- 需要登录才能发布
```

**步骤3：列举边界情况**
```
⚠️ 边界情况：
- 文章内容为空
- 文章标题超长
- 评论包含敏感词
- 用户重复提交
- 数据库连接失败
```

**步骤4：生成备选方案**
```
📝 备选方案：

方案A: 单体架构
- ✅ 简单、快速
- ❌ 后期扩展难
推荐理由：MVP阶段足够了
```

## 📊 功能开发模板

```markdown
## 功能： [名称]

### 🎯 核心需求
[一句话描述]

### ✅ 成功标准
- [ ] 标准1
- [ ] 标准2

### 📋 用户故事
- 作为 [角色]，我希望 [功能]，以便 [收益]

### ⚠️ 边界情况
| 场景 | 处理 |
|-----|------|
| 空数据 | 显示空状态 |
| 加载中 | Loading |
| 失败 | 错误+重试 |

### 🔧 技术约束
- 性能：<200ms
- 兼容：Chrome/Firefox
- 安全：XSS防护

### 📝 备选方案
[列出2-3个方案]

### ⏰ 估算
- MVP: X天
- 完整版: Y天
```

## 💡 苏格拉底式追问

深入理解的技巧：

### 1. 追问本质
```
"这个需求背后的真正目的是什么？"
"如果不做这个会怎样？"
```

### 2. 追问边界
```
"什么情况下这个方案会失败？"
"输入的极限是什么？"
```

### 3. 追问权衡
```
"选择A而不是B的最大理由是什么？"
"我们愿意牺牲什么来获得什么？"
```

### 4. 追问风险
```
"最坏的情况是什么？"
"如何快速恢复？"
```

## ⚠️ 常见陷阱

| ❌ 陷阱 | ✅ 正确做法 |
|--------|-----------|
| 上来就写代码 | 先明确需求 |
| 追求完美方案 | 先做MVP |
| 忽略边界情况 | 头脑风暴时列出 |
| 忽略性能 | 提前考虑扩展 |
| 单人决策 | 团队讨论 |

---

## ⏭️ 何时跳过

简单任务不需要完整头脑风暴：

- ✅ 简单的CRUD
- ✅ 已有的模式重复
- ✅ 紧急Bug修复
- ✅ 纯UI调整
- ✅ 小型重构

**判断标准**：如果10分钟内能想清楚，就不需要。
