## 🎯 这是什么？

**Systematic Debugging** 是一套系统化的调试方法论，帮助你用**4阶段流程**快速定位Bug根本原因，而不是盲目试错。

> "复杂性降低是主要目标" —— 大多数Bug源于复杂的解决方案，而非简单的错误。

---

## 🔄 4阶段调试流程

```
┌─────────────────────────────────────────────────────────┐
│  阶段1：假设形成                                          │
│  ↓ 理解问题，列出可能原因                                │
│  阶段2：证据收集                                          │
│  ↓ 隔离测试，对比实验                                     │
│  阶段3：根因分析                                          │
│  ↓ 5个为什么，鱼骨图分析                                  │
│  阶段4：修复验证                                          │
│  ↓ 测试覆盖，回归检查                                     │
└─────────────────────────────────────────────────────────┘
```

## 📝 使用示例

### 用户说："接口报500错误"

### AI按流程执行：

**阶段1：假设形成**
```
可能假设：
A. 数据库连接失败 - 上线后有过数据库切换
B. 请求参数格式错误 - 前端改了数据结构
C. 并发过高资源耗尽 - 最近流量增长
```

**阶段2：证据收集**
```bash
# 1. 查看错误日志
grep "500" app.log | tail -20

# 2. 创建最小复现用例
curl -X POST /api/data -d "固定测试数据"

# 3. 对比测试
# 回滚最近代码？测试是否通过？
```

**阶段3：根因分析**
```
为什么500？
→ 数据库连接超时
为什么超时？
→ 连接池10个连接全占用
为什么全占用？
→ 有一个慢查询（30秒）没释放
为什么慢？
→ 这张表缺少索引
根本原因：上线前没做性能测试
```

**阶段4：修复验证**
```sql
-- 添加索引
CREATE INDEX idx_xxx ON table_name(column);

-- 验证
EXPLAIN SELECT ...
```

## 🛠️ 常用调试命令

### 日志分析
```bash
# 实时查看错误
tail -f error.log | grep "关键词"

# 统计错误类型
grep -o "Error:.*" app.log | sort | uniq -c

# 按时间范围查找
sed -n '/2024-01-01 10:00/,/2024-01-01 11:00/p' app.log
```

### 代码调试
```python
# 添加诊断日志
import logging
logging.debug(f"Step: {step}, Data: {data}")

# 条件断点
if condition:
    import pdb; pdb.set_trace()
```

### 数据库调试
```sql
-- 查看慢查询
SHOW FULL PROCESSLIST;

-- 分析执行计划
EXPLAIN SELECT ...

-- 查看锁等待
SELECT * FROM information_schema.INNODB_LOCKS;
```

## ⚡ 常见Bug速查表

| 症状 | 可能原因 | 快速检查 |
|-----|---------|---------|
| NPE | 未初始化/null传入 | 搜索 `!= null` |
| 数组越界 | 边界计算错误 | 打印 length |
| 时区错误 | UTC/本地时间混淆 | 检查时区设置 |
| 乱码 | 编码不一致 | 检查 charset |
| 间歇失败 | 并发/缓存问题 | 加日志看频率 |
| OOM | 内存泄漏/大对象 | 看 heap dump |

## 📚 进阶技巧

### 5个为什么示例

```markdown
问题：用户反映提交订单很慢

为什么慢？→ 数据库写入慢
为什么慢？→ 有3个触发器在执行
为什么有触发器？→ 旧设计遗留
为什么要遗留？→ 没人知道它们干嘛的
为什么不知道？→ 没有文档

根本原因：缺少技术债务清理机制
```

### 二分查找定位法

```
代码有1000行，错误在第600行附近：
1. 先运行前500行 → 没报错 → 错误在后500行
2. 运行500-750行 → 报错 → 错误在500-625行
3. 运行500-562行 → 没报错 → 错误在562-625行
...
平均7步就能定位！
```

## ✅ 调试Checklist

- [ ] 复现问题（能否100%复现？）
- [ ] 收集证据（日志、堆栈、环境）
- [ ] 形成假设（至少3个可能原因）
- [ ] 隔离测试（最小复现场景）
- [ ] 找到根因（5个为什么）
- [ ] 实施修复（最小改动原则）
- [ ] 验证修复（能复现问题吗？）
- [ ] 回归测试（其他功能正常吗？）
- [ ] 添加监控（防止再次发生？）
- [ ] 更新文档（记录经验？）
